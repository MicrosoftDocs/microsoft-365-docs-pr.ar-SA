# <a name="microsoft-azure-architectures-for-sharepoint-2013"></a>Microsoft Azure Architectures SharePoint 2013

Azure هو بيئة جيدة لاستضافة حل SharePoint Server 2013. في معظم الحالات، نوصي Microsoft 365، ولكن قد SharePoint مزرعة خادم مستضافة في Azure خيارا جيدا لحلول معينة. تصف هذه المقالة كيفية تصميم حلول SharePoint بحيث تتناسب بشكل جيد مع نظام Azure الأساسي. يتم استخدام الحلين المحددين التاليين كأمثلة:
  
- [SharePoint خادم 2013 بعد الكوارث في Microsoft Azure](sharepoint-server-2013-disaster-recovery-in-microsoft-azure.md)
    
- [مواقع الإنترنت في Microsoft Azure SharePoint Server 2013](internet-sites-in-microsoft-azure-using-sharepoint-server-2013.md)
    
## <a name="recommended-sharepoint-solutions-for-azure-infrastructure-services"></a>حلول SharePoint الموصى بها لخدمات البنية الأساسية ل Azure

خدمات البنية الأساسية ل Azure هي خيار مقنع لاستضافة SharePoint الأساسية. بعض الحلول ملائمة لهذا النظام الأساسي بشكل أفضل من الحلول الأخرى. يعرض الجدول التالي الحلول المستحسنة.
  
|**الحل**|**لماذا يوصى باستخدام هذا الحل ل Azure**|
|:-----|:-----|
|بيئات التطوير والاختبار  <br/> |من السهل إنشاء هذه البيئات وإدارتها.  <br/> |
|استرداد مزرعة SharePoint في وقت SharePoint في Azure  <br/> |**مركز بيانات ثانوي مستضاف** استخدم Azure بدلا من الاستثمار في مركز بيانات ثانوي في منطقة مختلفة. <br/> **بيئات استرداد التكاليف المنخفضة لكارثات** الحفاظ على موارد أقل من بيئة استرداد الكوارث المحلية والدفع مقابلها. يعتمد عدد الموارد على بيئة استرداد الكوارث التي تختارها: الاستعداد الباردة أو الاستعداد الحار أو الاستعداد السريع. <br/> **نظام أساسي أكثر مرونة** في حالة حدوث كارثة، يمكنك بسهولة تغيير حجم مزرعة SharePoint لتلبية متطلبات التحميل. يمكنك المقياس عندما لا تحتاج إلى الموارد. <br/> راجع [SharePoint خادم 2013 بعد الكوارث في Microsoft Azure](sharepoint-server-2013-disaster-recovery-in-microsoft-azure.md).  <br/> |
|المواقع التي تواجه الإنترنت التي تستخدم ميزات ومقياس غير متوفرة في Microsoft 365  <br/> |**تركيز جهودك** ركز على إنشاء موقع رائع بدلا من إنشاء بنية أساسية. <br/> **الاستفادة من المرونة في Azure** يمكنك تحديد حجم المزرعة للطلب عن طريق إضافة خوادم جديدة، والدفع فقط مقابل الموارد التي تحتاج إليها. تخصيص الجهاز الديناميكي غير معتمد (المقياس التلقائي). <br/> **استخدام Azure Active Directory (AD)** استفد من Azure AD في حسابات العملاء. <br/> **أضف SharePoint الوظائف غير المتوفرة في Microsoft 365** إضافة تقارير وتحليلات ويب معمقة. <br/> راجع [مواقع الإنترنت في Microsoft Azure باستخدام SharePoint Server 2013](internet-sites-in-microsoft-azure-using-sharepoint-server-2013.md).  <br/> |
|مزارع التطبيقات لدعم Microsoft 365 أو البيئات المحلية  <br/> |**إنشاء التطبيقات واختبارها واستضافتها** في Azure لدعم البيئات المحلية والبيئات السحابية على حد سواء. <br/> **استضافة هذا الدور** في Azure بدلا من شراء أجهزة جديدة للبيئات المحلية. <br/> |
   
بالنسبة لحلول الإنترانت والتعاون وأحمال العمل، يجب التفكير في الخيارات التالية:
  
- حدد ما Microsoft 365 يلبي متطلبات عملك أو يمكن أن يكون جزءا من الحل. Microsoft 365 مجموعة ميزات غنية تكون دائما م تاريخا.
    
- إذا Microsoft 365 لا يلبي جميع متطلبات العمل، فنظر في تنفيذ قياسي SharePoint 2013 المحلي من Microsoft Consulting Services (MCS). يمكن أن تكون البنية القياسية حلا أسرع وأرخص وأسهل لدعمه من التصميم المخصص. 
    
- إذا لم يلبي التنفيذ القياسي متطلبات العمل الخاصة بك، ف فكر في حل مخصص في الموقع.
    
- إذا كان استخدام نظام أساسي سحابي مهما لمتطلبات العمل، فنظر في تنفيذ قياسي أو مخصص SharePoint 2013 المستضافة في خدمات البنية الأساسية في Azure. SharePoint هذه الحلول أسهل بكثير في Azure من الأنظمة الأساسية العامة الأخرى غير الأصلية في السحابة من Microsoft.
    
## <a name="before-you-design-the-azure-environment"></a>قبل تصميم بيئة Azure

في حين تستخدم هذه المقالة أمثلة SharePoint طبولوجيا، يمكنك استخدام مفاهيم التصميم هذه مع أي SharePoint للمزرعة. قبل تصميم بيئة Azure، استخدم الإرشادات التالية للطبولوجيا والهندسة والقدرة والأداء لتصميم المزرعة SharePoint:
  
- [تصميم هيكلي لمصممي SharePoint 2013 IT](/SharePoint/technical-reference/technical-diagrams)
    
- [التخطيط لإدارة الأداء والقدرات في SharePoint Server 2013](/SharePoint/administration/performance-planning-in-sharepoint-server-2013)
    
## <a name="determine-the-active-directory-domain-type"></a>تحديد نوع مجال Active Directory

تعتمد SharePoint الخادم على Active Directory لتوفير حسابات إدارية لإعداد المزرعة. في هذا الوقت، يوجد خياران SharePoint في Azure. يتم وصفها في الجدول التالي.
  
|**الخيار**|**الوصف**|
|:-----|:-----|
|مجال مخصص  <br/> |يمكنك نشر مجال Active Directory مخصص ومعزول إلى Azure لدعم SharePoint الخاصة بك. هذا خيار جيد لمواقع الإنترنت ذات الواجهة العامة.  <br/> |
|توسيع المجال في الموقع من خلال اتصال عبر الموقع  <br/> |عند توسيع المجال في الموقع من خلال اتصال داخل الموقع، يمكن للمستخدمين الوصول إلى مزرعة SharePoint عبر إنترانت كما لو كانت مستضافة في الموقع. يمكنك الاستفادة من تنفيذ Active Directory و DNS في الموقع.  <br/> الاتصال عبر الموقع مطلوب لبناء بيئة استرداد لكارثة في Azure للفشل في الوصول إلى من المزرعة المحلية.  <br/> |
   
تتضمن هذه المقالة مفاهيم التصميم لتوسيع المجال الداخلي من خلال اتصال داخل الموقع. إذا كان الحل يستخدم مجالا مخصصا، فلا تحتاج إلى اتصال عبر الموقع.
  
## <a name="design-the-virtual-network"></a>تصميم الشبكة الظاهرية

تحتاج أولا إلى شبكة ظاهرية في Azure، تتضمن شبكات فرعية سيتم وضع الأجهزة الظاهرية عليها. تحتاج الشبكة الظاهرية إلى مساحة عنوان IP خاصة، يتم تعيين أجزاء منها إلى الشبكات الفرعية.
  
إذا كنت تقوم بتوسيع شبكتك المحلية إلى Azure عبر اتصال عبر الموقع (مطلوب لبيئة استرداد في حالة حدوث كارثة)، فيجب اختيار مساحة عنوان خاصة غير موجودة بالفعل في مكان آخر في شبكة المؤسسة، والتي يمكن أن تتضمن بيئتك المحلية وشبكات Azure الظاهرية الأخرى. 
  
**الشكل 1: البيئة المحلية مع شبكة ظاهرية في Azure**

![تصميم شبكة Microsoft Azure الظاهرية SharePoint حل. شبكة فرعية واحدة لبوابة Azure. شبكة فرعية واحدة للآلات الظاهرية.](../media/OPrrasconWA-AZarch.png)
  
في هذا الرسم التخطيطي:
  
- يتم توضيح الشبكة الظاهرية في Azure جنبا إلى جنب مع البيئة المحلية. لم يتم توصيل البيئات بعد بواسطة اتصال عبر الموقع، والذي يمكن أن يكون اتصال VPN من موقع إلى موقع أو ExpressRoute.
    
- في هذه المرحلة، تتضمن الشبكة الظاهرية فقط الشبكات الفرعية ولا تتضمن أي عناصر معمارية أخرى. ستستضيف إحدى الشبكة الفرعية بوابة Azure وناشطات فرعية أخرى مستويات مزرعة SharePoint، مع واحدة إضافية ل Active Directory و DNS.
    
## <a name="add-cross-premises-connectivity"></a>إضافة اتصال عبر الموقع

وخطوة النشر التالية هي إنشاء الاتصال المحلي (إذا كان هذا ينطبق على الحل). بالنسبة إلى الاتصالات عبر الأماكن، توجد بوابة Azure في شبكة بوابة فرعية منفصلة، ويجب عليك إنشاء مساحة عنوان وتعيينها. 
  
عندما تخطط لاتصال عبر الموقع، يمكنك تعريف بوابة Azure وإنشاء اتصال بجهاز بوابة في الموقع.
  
**الشكل 2: استخدام بوابة Azure والجهاز الذي يعمل بالبوابة المحلية لتوفير اتصال موقع إلى موقع بين البيئة المحلية و Azure**

![البيئة المحلية المتصلة بشبكة Azure الظاهرية من خلال اتصال عبر الموقع، والذي يمكن أن يكون اتصال VPN من موقع إلى موقع أو ExpressRoute.](../media/AZarch-VPNgtwyconnct.png)
  
في هذا الرسم التخطيطي:
  
- بالإضافة إلى الرسم التخطيطي السابق، يتم توصيل البيئة المحلية بشبكة Azure الظاهرية من خلال اتصال عبر الموقع، والذي يمكن أن يكون اتصال VPN من موقع إلى موقع أو ExpressRoute.
    
- تكون بوابة Azure على شبكة بوابة فرعية.
    
- تتضمن البيئة المحلية جهاز بوابة، مثل موجه أو خادم VPN.
    
للحصول على معلومات إضافية حول التخطيط لشبكة ظاهرية داخل الموقع وإنشاءها، راجع الاتصال شبكة المحلية [إلى شبكة Microsoft Azure الظاهرية](connect-an-on-premises-network-to-a-microsoft-azure-virtual-network.md).
  
## <a name="add-active-directory-domain-services-ad-ds-and-dns"></a>إضافة خدمات مجال Active Directory (AD DS) وDNS

لاسترداد حالة الكوارث في Azure، يمكنك نشر Windows Server AD وDNS في سيناريو مختلط حيث يتم نشر Windows Server AD على أجهزة ظاهرية من Azure.
  
**الشكل 3: تكوين مجال Active Directory المختلط**

![إن أجهزة STwo الظاهرية التي تم نشرها على شبكة Azure الظاهرية وشبكة المزرعة الفرعية SharePoint هي وحدات تحكم بالمجالات المتماثلة وخادمات DNS.](../media/AZarch-HyADdomainConfig.png)
  
يتم إنشاء هذا الرسم التخطيطي على الرسومات التخطيطية السابقة عن طريق إضافة جهازين ظاهريين إلى Windows AD Server وDNS الفرعية. هذه الأجهزة الظاهرية هي وحدات تحكم المجال المتماثلة و خوادم DNS. وهي ملحق لبيئة خادم Windows المحلية. 
  
يوفر الجدول التالي توصيات التكوين لهذه الأجهزة الظاهرية في Azure. استخدمها كنقطة بداية لتصميم بيئتك الخاصة— حتى لمجال مخصص حيث لا تتواصل بيئة Azure مع بيئتك المحلية.
  
|**عنصر**|**تكوين**|
|:-----|:-----|
|حجم الجهاز الافتراضي في Azure  <br/> |الحجم A1 أو A2 في المستوى القياسي  <br/> |
|نظام التشغيل  <br/> |Windows Server 2012 R2  <br/> |
|دور Active Directory  <br/> |وحدة تحكم مجال AD DS المعينة كخادم كتالوج عام. يقلل هذا التكوين من حركة الخروج عبر الاتصال عبر الموقع.  <br/> في بيئة متعددة المجالات ذات معدلات تغيير عالية (هذا ليس شائعا)، قم بتكوين وحدات تحكم المجال في الموقع المحلي لكي لا تتم مزامنتها مع خوادم الكتالوجات العمومية في Azure، لتقليل نسبة تكرار البيانات.  <br/> |
|دور DNS  <br/> |قم بتثبيت خدمة خادم DNS وتكوينها على وحدات التحكم بالمجال.  <br/> |
|أقراص البيانات  <br/> |ضع قاعدة بيانات Active Directory والسجلات وSYSVOL على أقراص بيانات Azure إضافية. لا ضعها على قرص نظام التشغيل أو الأقراص المؤقتة التي يوفرها Azure.  <br/> |
|عناوين IP  <br/> |استخدم عناوين IP الثابتة وتكوين الشبكة الظاهرية لتعيين هذه العناوين إلى الأجهزة الظاهرية في الشبكة الظاهرية بعد تكوين وحدات التحكم بالمجال.  <br/> |
   
> [!IMPORTANT]
> قبل نشر Active Directory في Azure، اقرأ إرشادات نشر Windows [Server Active Directory على أجهزة Azure الظاهرية](/windows-server/identity/ad-ds/introduction-to-active-directory-domain-services-ad-ds-virtualization-level-100). تساعدك هذه على تحديد ما إذا كانت هناك حاجة إلى تصميم مختلف أو إعدادات تكوين مختلفة لحلك. 
  
## <a name="add-the-sharepoint-farm"></a>إضافة المزرعة SharePoint

ضع الأجهزة الظاهرية للمزرعة SharePoint في مستويات على الشبكات الفرعية المناسبة.
  
**الشكل 4: موضع SharePoint الظاهرية**

![خوادم قواعد البيانات SharePoint خوادم إضافية إلى شبكة Azure الظاهرية ضمن SharePoint الفرعية للمزرعة.](../media/AZarch-SPVMsinCloudSer.png)
  
يتم إنشاء هذا الرسم التخطيطي على الرسومات التخطيطية السابقة عن طريق SharePoint أدوار خادم المزرعة في مستوياتها الخاصة.
  
- جهازان ظاهريان يعملان SQL Server إنشاء طبقة قاعدة البيانات.
    
- جهازان ظاهريان يعملان SharePoint Server 2013 لكل من المستويات التالية: خوادم الواجهة الأمامية، خوادم ذاكرة التخزين المؤقت الموزعة، خوادم الخادم الخلفي.
    
## <a name="design-and-fine-tune-server-roles-for-availability-sets-and-fault-domains"></a>تصميم أدوار الخادم وضبطها  مجموعات التوفر ومجالات الخطأ

مجال الخطأ هو مجموعة من الأجهزة التي يتم تشغيل مثيلات الدور فيها. يمكن تحديث الأجهزة الظاهرية ضمن مجال الخطأ نفسه بواسطة البنية الأساسية Azure في الوقت نفسه. أو، قد تفشل في الوقت نفسه لأنها تشارك الحامل نفسه. لتجنب مخاطر وجود جهازين ظاهريين على مجال الخطأ نفسه، يمكنك تكوين الأجهزة الظاهرية كم مجموعة توفر، مما يضمن وجود كل جهاز ظاهري في مجال خطأ مختلف. إذا تم تكوين ثلاثة أجهزة ظاهرية كم مجموعة توفر، يضمن Azure عدم وجود أكثر من جهازين من الأجهزة الظاهرية في مجال الخطأ نفسه.
  
عند تصميم بنية Azure لمزرعة SharePoint، قم بتكوين أدوار خوادم متطابقة لتكون جزءا من مجموعة التوفر. يضمن ذلك انتشار الأجهزة الظاهرية عبر مجالات الخطأ المتعددة.
  
**الشكل 5: استخدام مجموعات توفر Azure لتوفير توفر عال SharePoint المزرعة**

![تكوين مجموعات التوفر في بنية Azure الأساسية لحل SharePoint 2013.](../media/AZenv-WinAzureAvailSetsHA.png)
  
يوضح هذا الرسم التخطيطي تكوين مجموعات التوفر ضمن بنية Azure الأساسية. يشارك كل واحد من الأدوار التالية مجموعة توفر منفصلة:
  
- Active Directory و DNS
    
- Database
    
- النهاية الخلفى
    
- توزيع ذاكرة التخزين المؤقت
    
- الواجهة الأمامية
    
قد SharePoint المزرعة إلى ضبط في نظام Azure الأساسي. لضمان توفر جميع المكونات بشكل عال، تأكد من تكوين أدوار الخادم كلها بشكل متطابق.
  
فيما يلي مثال يوضح بنية مواقع الإنترنت القياسية التي تلبي أهداف الأداء والقدرات المحددة. يتم عرض هذا المثال في نموذج الهندسة التالي: هندسة البحث في مواقع الإنترنت [SharePoint Server 2013](https://go.microsoft.com/fwlink/p/?LinkId=261519).
  
**الشكل 6: مثال تخطيطي لتحقيق أهداف القدرة الإنتاجية والأداء في مزرعة من ثلاث مستويات**

![تصميم SharePoint الإنترنت القياسية في 2013 مع تخصيصات المكونات التي تلبي أهداف الأداء والقدرات المحددة.](../media/AZarch-CapPerfexmpArch.png)
  
في هذا الرسم التخطيطي:
  
- يتم تمثيل مزرعة من ثلاث مستويات: خوادم الويب، خوادم التطبيقات، خوادم قواعد البيانات.
    
- يتم تكوين خوادم الويب الثلاثة بشكل متطابق مع مكونات متعددة.
    
- يتم تكوين خادمي قاعدة البيانات بشكل متطابق.
    
- لا يتم تكوين خوادم التطبيقات الثلاثة بشكل متطابق. تتطلب أدوار الخادم هذه ضبط مجموعات التوفر في Azure.
    
دعنا نلق نظرة عن قرب على مستوى خادم التطبيق.
  
**الشكل 7: طبقة خادم التطبيق قبل ضبطها**

![مثال SharePoint خادم تطبيق Server 2013 قبل ضبط مجموعات توفر Microsoft Azure.](../media/AZarch-AppServtierBefore.png)
  
في هذا الرسم التخطيطي:
  
- يتم تضمين ثلاثة خوادم في طبقة التطبيق.
    
- يتضمن الخادم الأول أربعة مكونات.
    
- يتضمن الخادم الثاني ثلاثة مكونات.
    
- يتضمن الخادم الثالث مكونين.
    
يمكنك تحديد عدد المكونات حسب أهداف القدرة الإنتاجية والأداء للمزرعة. لتكييف هذه البنية مع Azure، سنقوم بتكرور المكونات الأربعة عبر الخوادم الثلاثة كلها. يؤدي ذلك إلى زيادة عدد المكونات التي تتجاوز ما هو ضروري للأداء والقدرة. المبادلات هي أن هذا التصميم يضمن توفرا عاليا للمكونات الأربعة كلها في نظام Azure الأساسي عند تعيين هذه الأجهزة الظاهرية الثلاثة إلى مجموعة توفر.
