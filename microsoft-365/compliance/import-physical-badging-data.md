# <a name="set-up-a-connector-to-import-physical-badging-data-preview"></a>إعداد موصل لاستيراد بيانات غير مهيئة (معاينة)

يمكنك إعداد موصل بيانات في مركز التوافق في Microsoft 365 لاستيراد بيانات غير مهيئة، مثل أحداث الوصول الفعلي الخام للموظف أو أي منبهات وصول فعلي تم إنشاؤها بواسطة نظام الازد الوارد في مؤسستك. من أمثلة نقاط الوصول الفعلية إدخال إلى مبنى أو إدخال إلى غرفة الخادم أو مركز البيانات. يمكن استخدام بيانات الضار الفعلية بواسطة حل إدارة المخاطر Microsoft 365 [insider](insider-risk-management.md) للمساعدة في حماية مؤسستك من النشاط الضار أو سرقة البيانات داخل مؤسستك.

تتألف عملية إعداد موصل ا غير موصل فعلي من المهام التالية:

- إنشاء تطبيق في Azure Active Directory (Azure AD) للوصول إلى نقطة نهاية API تقبل تحميل JSON يحتوي على بيانات تدبير فعلي.

- إنشاء تحميل JSON باستخدام مخطط معرف بواسطة موصل بيانات تالف فعلي.

- إنشاء موصل بيانات غير موصل فعلي في مركز التوافق في Microsoft 365.

- تشغيل برنامج نصي لدفع البيانات الفعلية إلى نقطة نهاية API.

- بشكل اختياري، يتم جدولة البرنامج النصي لكي يتم تشغيله تلقائيا لاستيراد بيانات تدخير البيانات الفعلية حاليا.

## <a name="before-you-set-up-the-connector"></a>قبل إعداد الموصل

- يجب أن يتم تعيين دور مسؤول موصل البيانات للمستخدم الذي ينشئ موصل الازدئ الفعلي في الخطوة 3. هذا الدور مطلوب لإضافة موصلات على صفحة **موصلات** البيانات في مركز التوافق في Microsoft 365. يضاف هذا الدور بشكل افتراضي إلى مجموعات دور متعددة. للحصول على قائمة لمجموعات الأدوار هذه، راجع القسم "الأدوار في مراكز الأمان والتوافق" في الأذونات [في مركز التوافق & الأمان](../security/office-365-security/permissions-in-the-security-and-compliance-center.md#roles-in-the-security--compliance-center). بدلا من ذلك، يمكن للمسؤول في مؤسستك إنشاء مجموعة دور مخصصة وتعيين دور مسؤول موصل البيانات ثم إضافة المستخدمين المناسبين كأعضاء. للحصول على الإرشادات، راجع المقطع "إنشاء مجموعة دور مخصص" في [الأذونات في مركز التوافق في Microsoft 365](microsoft-365-compliance-center-permissions.md#create-a-custom-role-group).

   > [!NOTE]
   > دور مسؤول موصل البيانات غير معتمد حاليا في بيئات سحابة القطاع الحكومي و DoD العالية. لذلك، يجب تعيين دور استيراد تصدير علبة البريد في سحابة القطاع الحكومي إلى المستخدم الذي يقوم بإنشاء موصل الموارد البشرية في سحابة القطاع الحكومي عالية و Do Exchange Online D. بشكل افتراضي، لا يتم تعيين هذا الدور إلى أي مجموعة دور في Exchange Online. يمكنك إضافة دور استيراد تصدير علبة البريد إلى مجموعة دور إدارة المؤسسة في Exchange Online. أو يمكنك إنشاء مجموعة دور جديدة، وتعيين دور استيراد تصدير علبة البريد، ثم إضافة المستخدمين المناسبين كأعضاء. لمزيد من المعلومات، راجع المقاطع [إنشاء](/Exchange/permissions-exo/role-groups#create-role-groups) مجموعات الدور أو [](/Exchange/permissions-exo/role-groups#modify-role-groups) تعديل مجموعات الدور في المقالة "إدارة مجموعات الدور في Exchange Online".

- تحتاج إلى تحديد كيفية استرداد البيانات أو تصديرها من نظام التهيئة الفعلي الخاص بالم مؤسستك (بشكل يومي) وإنشاء ملف JSON تم وصفه في الخطوة 2. سيدفع البرنامج النصي الذي تقوم بتشغيله في الخطوة 4 البيانات الموجودة في ملف JSON إلى نقطة نهاية API.

- يدفع البرنامج النصي النموذجي الذي تقوم بتشغيله في الخطوة 4 بيانات التهدير الفعلية من ملف JSON إلى API للموصل بحيث يمكن استخدامه بواسطة حل إدارة مخاطر insider. هذا البرنامج النصي النموذجي غير معتمد ضمن أي برنامج أو خدمة دعم قياسية من Microsoft. يتم توفير البرنامج النصي النموذجي AS IS بدون ضمان من أي نوع. وتنقر Microsoft أيضا عن جميع الضمانات الضمنية، بما في ذلك، على سبيل المثال لا الحصر، أي ضمانات ضمنية ل قابلية الاستخدام أو اللياقة لغرض معين. يبقى الخطر بأكمله الناتج عن استخدام أو أداء البرنامج النصي النموذجي والوثائق معك. لا تتحمل Microsoft بأي حال من الأحوال، أو كتابها، أو أي شخص آخر يشارك في إنشاء البرامج النصية أو إنتاجها أو تسليمها المسؤولية عن أي أضرار أيا كانت (بما في ذلك، على سبيل المثال لا الحصر، الأضرار الناجمة عن خسارة أرباح الأعمال أو انقطاع الأعمال أو فقدان معلومات العمل أو أي خسارة مادية أخرى) الناشئة عن استخدام البرامج النصية أو الوثائق العينة أو عدم القدرة على استخدامها،  حتى لو تم نصح Microsoft بإمكانية حدوث مثل هذه الأضرار.

- يتوفر هذا الموصل في بيئات سحابة القطاع الحكومي في Microsoft 365 الحكومة الأمريكية. قد تشتمل التطبيقات والخدمات التي تقدمها جهة خارجية على تخزين بيانات عملاء مؤسستك ونقلها وحمايتها على أنظمة أخرى تقع خارج البنية الأساسية ل Microsoft 365 وبالتالي لا تغطيها التزامات التوافق وحماية البيانات Microsoft 365. لا تقدم Microsoft أي تمثيل يشير فيه استخدام هذا المنتج للاتصال إلى تطبيقات  الأطراف الخارجية إلى أن تلك التطبيقات التي تقدمها جهة خارجية متوافقة مع FEDRAMP.

## <a name="step-1-create-an-app-in-azure-active-directory"></a>الخطوة 1: إنشاء تطبيق في Azure Active Directory

الخطوة الأولى هي إنشاء تطبيق جديد وتسجيله في Azure Active Directory (Azure AD). سيتوافق التطبيق مع موصل التهيئة الفعلي الذي تقوم بإنشاءه في الخطوة 3. سيتيح إنشاء هذا التطبيق ل Azure AD مصادقة طلب الدفع الخاص بالحمل الفعلي ل JSON الذي يحتوي على بيانات غير صحيحة. أثناء إنشاء تطبيق Azure AD هذا، تأكد من حفظ المعلومات التالية. سيتم استخدام هذه القيم في الخطوات اللاحقة.

- Azure AD application ID (يسمى أيضا "اسم التطبيق *"* أو *"اسم العميل"*)

- سرية تطبيق Azure AD (تسمى أيضا *سرية العميل*)

- "اسم المستأجر" (يسمى أيضا *"اسم الدليل"*)

للحصول على إرشادات مفصلة خطوة بخطوة لإنشاء تطبيق في Azure AD، راجع تسجيل [تطبيق باستخدام](/azure/active-directory/develop/quickstart-register-app) النظام الأساسي للهويات في Microsoft.

## <a name="step-2-prepare-a-json-file-with-physical-badging-data"></a>الخطوة 2: إعداد ملف JSON مع بيانات تدنس فعلي

الخطوة التالية هي إنشاء ملف JSON يحتوي على معلومات حول بيانات الوصول الفعلي للموظفين. كما هو موضح في المقطع قبل البدء، ستحتاج إلى تحديد كيفية إنشاء ملف JSON هذا من نظام التهتك الفعلي في مؤسستك.

يجب أن يتوافق ملف JSON مع تعريف المخطط الذي يتطلبه الموصل. فيما يلي أوصاف لخصائص المخطط المطلوبة لملف JSON:

|الخاصية|الوصف|نوع البيانات|
|---|---|---|
|UserId|يمكن أن يكون للموظف هويات رقمية متعددة عبر الأنظمة. يجب أن يكون لدى الإدخال بالفعل Azure AD ID تم حله بواسطة النظام المصدر.|UPN أو عنوان البريد الإلكتروني|
|AssetId|المرجع المرجع للأصول المادية أو نقطة الوصول الفعلية.|سلسلة Alphanumeric|
|AssetName|الاسم الصديق للأصول المادية أو نقطة الوصول الفعلية.|سلسلة Alphanumeric|
|EventTime|الطابع الزمني للوصول.|التاريخ والوقت، بتنسيق UTC|
|AccessStatus|قيمة أو `Success``Failed`|سلسلة|
|||

فيما يلي مثال لملف JSON يتوافق مع المخطط المطلوب:

```json
[
    {
        "UserId":"sarad@contoso.com",
        "AssetId":"Mid-Sec-7",
        "AssetName":"Main Building 1st Floor Mid Section",
        "EventTime":"2019-07-04T01:57:49",
        "AccessStatus":"Failed"
    },
    {
        "UserId":"pilarp@contoso.com",
        "AssetId":"Mid-Sec-7",
        "AssetName":"Main Building 1st Floor Mid Section",
        "EventTime":"2019-07-04T02:57:49",
        "AccessStatus":"Success"
    }
]
```

يمكنك أيضا تنزيل تعريف المخطط التالي لملف JSON من المعالج عند إنشاء موصل تالف فعلي في الخطوة 3.

```text
{
   "title" : "Physical Badging Signals",
   "description" : "Access signals from physical badging systems",
   "DataType" : {
      "description" : "Identify what is the data type for input signal",
      "type" : "string",
   },
   "type" : "object",
   "properties": {
      "UserId" : {
         "description" : "Unique identifier AAD Id resolved by the source system",
         "type" : "string",
      },
      "AssetId": {
         "description" : "Unique ID of the physical asset/access point",
         "type" : "string",
      },
      "AssetName": {
         "description" : "friendly name of the physical asset/access point",
         "type" : "string",
      },
      "EventTime" : {
         "description" : "timestamp of access",
         "type" : "string",
      },
      "AccessStatus" : {
         "description" : "what was the status of access attempt - Success/Failed",
         "type" : "string",
      },
   }
   "required" : ["UserId", "AssetId", "EventTime" "AccessStatus"]
}
```

## <a name="step-3-create-the-physical-badging-connector"></a>الخطوة 3: إنشاء موصل الازديج الفعلي

الخطوة التالية هي إنشاء موصل فعلي للتهيئ في مركز التوافق في Microsoft 365. بعد تشغيل البرنامج النصي في الخطوة 4، تتم معالجة ملف JSON الذي أنشأته في الخطوة 3 ويدفع إلى نقطة نهاية API التي قمت بتكوينها في الخطوة 1. في هذه الخطوة، تأكد من نسخ «المهى الوظيفي» الذي يتم إنشاؤه عند إنشاء الموصل. سوف تستخدم "المعرف الوظيفي" عند تشغيل البرنامج النصي.

1. انتقل إلى مركز التوافق في Microsoft 365، وحدد <a href="https://go.microsoft.com/fwlink/p/?linkid=2173865" target="_blank">**موصلات البيانات**</a>.

2. على الصفحة **موصلات البيانات** ضمن **الازدوج الفعلي**، انقر فوق **عرض**.

3. في صفحة **الازدوج** الفعلي، انقر فوق **إضافة موصل**.

4. في صفحة **بيانات اعتماد المصادقة** ، قم بما يلي، ثم انقر فوق **التالي**:

   1. اكتب أو اللصق في "معرّف تطبيق Azure AD" لتطبيق Azure الذي أنشأته في الخطوة 1.

   2. قم بتنزيل المخطط النموذجي للمرجع الخاص بك لإنشاء ملف JSON.

   3. اكتب اسما فريدا لموصل الازديج الفعلي.

5. في الصفحة **مراجعة** ، راجع الإعدادات، ثم انقر فوق **إنهاء** لإنشاء الموصل.

6. يتم عرض صفحة الحالة التي تؤكد إنشاء الموصل. تحتوي هذه الصفحة أيضا على "معرّف المهمة". يمكنك نسخ "معرِّف المهمة" من هذه الصفحة أو من صفحة "النسخة المنسخة" للموصل. أنت بحاجة إلى "معر ة المهمة" هذه عند تشغيل البرنامج النصي.

   تحتوي صفحة الحالة أيضا على ارتباط إلى البرنامج النصي. راجع هذا البرنامج النصي لفهم كيفية نشر ملف JSON إلى نقطة نهاية API.

7. انقر **فوق تم**.

   يتم عرض الموصل الجديد في القائمة على علامة **التبويب موصلات** .

8. انقر فوق موصل الازديج الفعلي الذي أنشأته للتو لعرض صفحة القائمة من القائمة الإعلانية، التي تحتوي على خصائص ومعلومات أخرى حول الموصل.

## <a name="step-4-run-the-script-to-post-your-json-file-containing-physical-badging-data"></a>الخطوة 4: تشغيل البرنامج النصي من أجل نشر ملف JSON الذي يحتوي على بيانات تدنس فعلي

الخطوة التالية في إعداد موصل تدبير فعلي هي تشغيل برنامج نصي سيدفع بيانات التهيئة الفعلية في ملف JSON (الذي أنشأته في الخطوة 2) إلى نقطة نهاية API التي أنشأتها في الخطوة 1. نحن نوفر برنامج نصي عينة لمرجعك، كما يمكنك اختيار استخدامه أو إنشاء برنامج نصي خاص بك ل نشر ملف JSON إلى نقطة نهاية API.

بعد تشغيل البرنامج النصي، يتم دفع ملف JSON الذي يحتوي على البيانات الفعلية التي تسيء استخدام البيانات إلى Microsoft 365 حيث يمكن الوصول إليه بواسطة حل إدارة مخاطر insider. نوصي بنشر بيانات التهتك الفعلية يوميا. يمكنك القيام بذلك من خلال أتمتة العملية لإنشاء ملف JSON كل يوم من نظام الازدغة الفعلي ثم جدولة البرنامج النصي لدفع البيانات.

> [!NOTE]
> إن الحد الأقصى لعدد السجلات في ملف JSON الذي يمكن معالجته بواسطة API هو 50000 سجل.

1. انتقل إلى [هذا GitHub للوصول](https://github.com/microsoft/m365-physical-badging-connector-sample-scripts/blob/master/push_physical_badging_records.ps1) إلى البرنامج النصي النموذجي.

2. انقر فوق **الزر "الخام** " لعرض البرنامج النصي في طريقة عرض النص

3. انسخ كل الأسطر في البرنامج النصي النموذجي ثم احفظها في ملف نصي.

4. قم بتعديل البرنامج النصي النموذجي لمنظمتك، إذا لزم الأمر.

5. احفظ الملف النصي كملف نصي Windows PowerShell باستخدام لاحقة اسم الملف .ps1، على سبيل المثال، PhysicalBadging.ps1.

6. افتح موجه الأوامر على الكمبيوتر المحلي، واذهب إلى الدليل حيث حفظت البرنامج النصي.

7. تشغيل الأمر التالي لدفع بيانات التهدير الفعلية في ملف JSON إلى سحابة Microsoft؛ على سبيل المثال:

   ```powershell
   .\PhysicalBadging.ps1 -tenantId "<Tenant Id>" -appId "<Azure AD App Id>" -appSecret "<Azure AD App Secret>" -jobId "Job Id" -jsonFilePath "<records file path>"
   ```

   يصف الجدول التالي المعلمات التي يجب استخدامها مع هذا البرنامج النصي وقيمها المطلوبة. يتم استخدام المعلومات التي حصلت عليها في الخطوات السابقة في قيم هذه المعلمات.

   |المعلمة|الوصف|
   |---|---|
   |tenantId|هذا هو الم ID الخاص Microsoft 365 الذي حصلت عليه في الخطوة 1. يمكنك أيضا الحصول على المستأجرمزيد لمنظمتك على **ريشة النظرة** العامة في مركز إدارة Azure AD. يتم استخدام هذا لتحديد مؤسستك.|
   |appId|هذا هو "معر ة تطبيق Azure AD" للتطبيق الذي أنشأته في Azure AD في الخطوة 1. يستخدم هذا بواسطة Azure AD للمصادقة عندما يحاول البرنامج النصي الوصول إلى Microsoft 365 مؤسستك.|
   |appSecret|هذا هو سر تطبيق Azure AD للتطبيق الذي أنشأته في Azure AD في الخطوة 1. يستخدم هذا أيضا للمصادقة.|
   |jobId|هذا هو "الماجد الوظيفي" لموصل التهيئة الفعلي الذي أنشأته في الخطوة 3. يتم استخدام هذا الأمر لاقران بيانات الازدوج الفعلية التي يتم دفعها إلى سحابة Microsoft بموصل الازدوج الفعلي.|
   |JsonFilePath|هذا هو مسار الملف على الكمبيوتر المحلي (الذي تستخدمه لتشغيل البرنامج النصي) لملف JSON الذي أنشأته في الخطوة 2. يجب أن يتبع هذا الملف المخطط النموذجي الموضح في الخطوة 3.|
   |||

   فيما يلي مثال على بناء جملة البرنامج النصي لموصل تالف فعلي باستخدام القيم الفعلية لكل معلمة:

   ```powershell
   .\PhysicalBadging.ps1 -tenantId d5723623-11cf-4e2e-b5a5-01d1506273g9 -appId 29ee526e-f9a7-4e98-a682-67f41bfd643e -appSecret MNubVGbcQDkGCnn -jobId b8be4a7d-e338-43eb-a69e-c513cd458eba -jsonFilePath 'C:\Users\contosoadmin\Desktop\Data\physical_badging_data.json'
   ```

   إذا نجح التحميل، يعرض البرنامج النصي Upload **بنجاح**.

   إذا كان لديك ملفات JSON متعددة، يجب تشغيل البرنامج النصي لكل ملف.

> [!NOTE]
> يمكنك أيضا اختيار نقل بيانات التهدير الفعلية إلى نقطة نهاية API باستخدام أساليب أخرى غير تشغيل البرنامج النصي السابق. على سبيل المثال، إليك نموذج لاستخدام Postman لدفع البيانات إلى نقطة نهاية API.

## <a name="step-5-monitor-the-physical-badging-connector"></a>الخطوة 5: مراقبة موصل الازديج الفعلي

بعد إنشاء موصل الازدحم الفعلي ودفع بيانات التهيئة الفعلية، يمكنك عرض الموصل ووضع التحميل في مركز التوافق في Microsoft 365. إذا قمت لجدولة تشغيل البرنامج النصي تلقائيا بشكل منتظم، يمكنك أيضا عرض الحالة الحالية بعد آخر مرة تم فيها تشغيل البرنامج النصي.

1. انتقل إلى مركز التوافق في Microsoft 365، وحدد <a href="https://go.microsoft.com/fwlink/p/?linkid=2173865" target="_blank">**موصلات البيانات**</a>.

2. انقر فوق **علامة التبويب موصلات** ، ثم حدد الموصل الذي يؤدي إلى التهزئ لعرض صفحة منتحلة. تحتوي هذه الصفحة على الخصائص والمعلومات حول الموصل.

   ![صفحة منتحلة عن الحالة لموصل الازديج الفعلي.](..\media\PhysicalBadgingStatusFlyout.png)

3. ضمن **الاستيراد الأخير**، انقر فوق الارتباط **تنزيل السجل** لفتح (أو حفظ) سجل الحالة للموصل. يحتوي هذا السجل على معلومات حول كل مرة يتم فيها تشغيل البرنامج النصي وتحميل البيانات من ملف JSON إلى سحابة Microsoft.

   ![يعرض ملف سجل موصل التهتك الفعلي عدد العناصر من ملف JSON الذي تم تحميله.](..\media\PhysicalBadgingConnectorLogFile.png)

   يشير **الحقل RecordsSaved** إلى عدد العناصر في ملف JSON الذي تم تحميله. على سبيل المثال، إذا كان ملف JSON يحتوي على أربعة كائنات، فإن قيمة الحقول **RecordsSaved** هي 4، إذا قام البرنامج النصي بتحميل كافة العناصر الموجودة في ملف JSON بنجاح.

إذا لم تقم بتشغيل البرنامج النصي في الخطوة 4، يتم عرض ارتباط لتنزيل البرنامج النصي ضمن **الاستيراد الأخير**. يمكنك تنزيل البرنامج النصي ثم اتباع الخطوات في الخطوة 4 لتشغيله.

## <a name="optional-step-6-schedule-the-script-to-run-automatically"></a>(اختياري) الخطوة 6: جدولة البرنامج النصي للتشغيل تلقائيا

للتأكد من توفر أحدث بيانات سوء استخدام مادية من مؤسستك لأدوات مثل حل إدارة مخاطر insider، نوصيك لجدولة البرنامج النصي لكي يتم تشغيله تلقائيا بشكل متكرر، مثل مرة واحدة في اليوم. يتطلب هذا الأمر أيضا تحديث بيانات التهتك الفعلية إلى ملف JSON في جدول مماثل (إن لم يكن هو نفسه) بحيث يحتوي على أحدث المعلومات حول الموظفين الذين يغادرون مؤسستك. الهدف من ذلك هو تحميل أحدث بيانات الازدحم الفعلية بحيث يمكن لموصل الازدحم الفعلي جعله متوفرا لحل إدارة مخاطر insider.

يمكنك استخدام تطبيق جدولة المهام في Windows لتشغيل البرنامج النصي تلقائيا كل يوم.

1. على الكمبيوتر المحلي، انقر فوق الزر Windows **البدء** ثم اكتب **جدول المهام**.

2. انقر فوق **تطبيق جدولة** المهام لفتحه.

3. في المقطع **إجراءات** ، انقر فوق **إنشاء مهمة**.

4. على علامة **التبويب عام** ، اكتب اسما وصفيا للمهمة المجدولة؛ على سبيل المثال **، برنامج نصي للموصل الذي يسيء إلى موصل فعلي**. يمكنك أيضا إضافة وصف اختياري.

5. ضمن **خيارات الأمان**، يمكنك القيام بما يلي:

   1. حدد ما إذا كنت تريد تشغيل البرنامج النصي فقط عند تسجيل الدخول إلى الكمبيوتر أو تشغيله عند تسجيل الدخول أم لا.

   2. تأكد **من تحديد** خانة الاختيار تشغيل بأعلى امتيازات.

6. حدد علامة **التبويب المشغلات** ، **وانقر فوق جديد**، ثم قم بما يلي:

   1. ضمن **الإعدادات**، حدد الخيار **يوميا**، ثم اختر تاريخا وتاريخا لتشغيل البرنامج النصي للمرة الأولى. سيتم تشغيل البرنامج النصي كل يوم في الوقت المحدد نفسه.

   2. ضمن **إعدادات متقدمة**، تأكد من تحديد خانة  الاختيار تمكين.

   3. انقر **فوق موافق**.

7. حدد علامة **التبويب** إجراءات، **وانقر فوق جديد**، ثم قم بما يلي:

   ![إعدادات الإجراء لإنشاء مهمة مجدولة جديدة للنص النصي للموصل الذي يسيء استخدام النص الفعلي.](..\media\SchedulePhysicalBadgingScript1.png)

   1. في القائمة **المنسدلة** إجراء، تأكد من **تحديد بدء برنامج** .

   2. في المربع **برنامج/برنامج نصي**، انقر فوق استعراض، واذهب إلى الموقع التالي وحدده بحيث يتم عرض المسار في المربع: C:\Windows\System32\WindowsPowerShell\v1.0\powershell.exe.

   3. في المربع **إضافة وسيطات (اختياري)،** قم بلصق الأمر البرنامج النصي نفسه الذي قمت بركضه في الخطوة 4. على سبيل المثال، .\PhysicalBadging.ps1-tenantId "d5723623-11cf-4e2e-b5a5-01d1506273g9" -appId "c12823b7-b55a-4989-faba-02de41bb97c3" -appEcret "MNubVGbcQDkGCnn" -jobId "e081f4f4-3831-48d6-7bb3-fcfab1581458" -jsonFilePath "C:\Users\contosoadmin\Desktop\Data\physical_badging_data.json"

   4. في **المربع البدء في (اختياري)،** اللصق في موقع مجلد البرنامج النصي الذي قمت بركضه في الخطوة 4. على سبيل المثال، C:\Users\contosoadmin\Desktop\Scripts.

   5. انقر **فوق** موافق لحفظ إعدادات الإجراء الجديد.

8. في النافذة **إنشاء مهمة** ، انقر فوق **موافق** لحفظ المهمة المجدولة. قد يتم مطالبتك بإدخال بيانات اعتماد حساب المستخدم.

   يتم عرض المهمة الجديدة في مكتبة جدولة المهام.

   ![يتم عرض المهمة الجديدة في مكتبة جدولة المهام.](..\media\SchedulePhysicalBadgingScript2.png)

في المرة الأخيرة التي تم فيها تشغيل البرنامج النصي، وفي المرة التالية التي يتم فيها جدولة تشغيله، يتم عرض البرنامج النصي. يمكنك النقر نقرا مزدوجا فوق المهمة لتحريرها.

يمكنك أيضا التحقق من المرة الأخيرة التي تم فيها تشغيل البرنامج النصي على صفحة منتحل البيانات لموصل الازديج الفعلي المناظر في مركز التوافق.
